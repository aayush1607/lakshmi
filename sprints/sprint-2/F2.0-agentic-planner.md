# F2.0 — Agentic Planner Loop

## One-liner
The agent picks its own tools per question, instead of running the same three every time.

## User story
> As a user asking "why is HUL down?", I want Lakshmi to figure out *on its own* that it needs the current quote, a week of history, sector peers, and recent filings — and to actually fetch them — instead of refusing because it only ever looks at my holdings.

## Expected behaviour (PM view)
- The agent plans before it fetches. For a typical question the user sees (in the `/why` trace, never the answer):
  ```
  plan   → decided to call: quote(HINDUNILVR), historical(HINDUNILVR, 30d),
                               index_quote(NIFTYFMCG), corporate_announcements(HINDUNILVR, 30d)
  fetch  → 4 tools in parallel · 812 ms
  reason → verdict drafted, 3 sources cited
  ```
- If the model's confidence is low *or* it explicitly requests more data, the loop runs one more planner iteration (capped at 3 total). The user sees this reflected in the trace, not the answer.
- Tool failures are **surfaced, not swallowed**. If `corporate_announcements` returns empty or errors, the model sees "no filings found" as a fact and says so in a Why bullet — instead of silently refusing.
- When two tools give conflicting numbers (Kite says ₹2,412 · NSE says ₹2,410.50), the verdict is yellow and both values appear with their sources.
- Every answer still obeys the F1.6 shape. The loop is invisible in the happy path.
- For trivial lookups ("what's my total invested amount?") the planner is allowed to pick just `portfolio_holdings` and exit the loop in one pass. No wasted calls.

## Acceptance criteria
1. For "why is HUL down?" the agent calls `symbol_resolve` + at least two of {`quote`, `historical`, `peer_compare`, `corporate_announcements`, `news_search`} — no hand-coded plan.
2. For "what's my total invested amount?" the agent calls exactly one tool (`portfolio_holdings`). No redundant fetches.
3. If a chosen tool errors, the error appears as a Why bullet with its source labelled `tool:error` — the answer is never silently reduced to "I don't have a source".
4. The total loop is bounded: max 3 planner iterations, max 8 tool calls, hard 30s wallclock. Exceeding any of these produces a refusal with a specific reason ("ran out of iterations", not "no source").
5. Two-source conflicts produce a yellow verdict and render *both* values with citations.
6. The planner's tool-choice JSON is validated against the registered tool schemas before any tool runs — a hallucinated tool name never reaches the fetch step.
7. `/why` shows per-iteration what was planned, what was fetched, and how long each step took.

## 🏰 Moat contribution
This is the single biggest reliability lever in the product. Hardcoded plans cap the ceiling of what Lakshmi can answer; a planner loop lets every new tool we add (F2.8, and later) immediately expand the set of answerable questions without a code change to the agent. Combined with Tier-1 sources, this is what turns "grounded chat" from a feature into a discipline.

## Out of scope
- Streaming tool output back into the model mid-execution (tools still run to completion before the next step).
- Human-in-the-loop tool approval (every tool is read-only in Sprint 2; approval comes with the first write tool).
- Learned tool routing / fine-tuning — the planner is pure prompt + JSON schema, no training data.
- Multi-turn tool conversations (e.g. the model asking a tool a follow-up question). One call per tool per iteration.

## Dependencies
- F1.5 (Grounded ask — the reason step already exists)
- F1.6 (Response shaper — verdict + confidence are already typed)

## Demoable in
Run `/ask why is HUL down` on an account that holds HUL. Show `/why` afterwards. Compare with a direct "what's my total invested?" and point out the single-tool-call efficiency.
