# F3.1 — Mantra Engine + Built-in Playbooks

## One-liner
A **Mantra** is a YAML-defined playbook Lakshmi can run on demand — and we ship 3 useful ones out of the box.

## User story
> As a user who repeats the same 4–5 checks every morning, I want to codify them once and run them as a single command, so my routine is one keystroke.

## Expected behaviour (PM view)
- User can list available mantras: `/mantra list`.
- 3 built-ins ship by default:
  - **morning-brief** — market overview + my portfolio P&L + top 3 news items on my holdings.
  - **watchlist-scan** — run `/ta` and `/fa` across a user-defined watchlist; flag anything notable.
  - **quarterly-review** — last quarter: what's up/down in my portfolio, why, and what I should watch next.
- User runs `/mantra run morning-brief`.
- Output is a sequence of verdict-formatted blocks (one per step), followed by a **Summary** verdict at the top.
- User can see / copy / edit any mantra: `/mantra show morning-brief` prints the YAML.
- User can create a new one from scratch with `/mantra new <name>` — opens `$EDITOR` with a template.
- Mantras live in `~/.lakshmi/mantras/*.yaml`.

## 🏰 Moat contribution
Mantras are the product's **workflow lock-in**. A user's accumulated playbooks — morning brief, watchlist logic, custom checks — become their personal investing OS and their switching cost. A raw LLM has no equivalent. The moat only materialises if the YAML is expressive enough to be worth keeping, so err toward capability over simplicity in the schema.

## Acceptance criteria
1. Each built-in mantra runs end-to-end and produces a grounded, formatted report.
2. Running the same mantra twice gives deterministic output (same structure, updated numbers).
3. Editing a YAML file and re-running picks up the changes — no restart needed.
4. A malformed YAML produces a clear error with line number, not a stack trace.
5. Mantras can call any command the user could type (`/ta`, `/fa`, `/portfolio health`, `ask …`).

## Out of scope
- Mantra marketplace / sharing (v2).
- Scheduled auto-runs (that's F3.4 — Aartis).
- Conditional branching / loops inside mantras (keep v1 linear).
- Hindi / regional mantras (v1.5).

## Dependencies
- F1.6 (Response shaper) for consistent sub-outputs.
- F2.* (TA, FA, portfolio) — mantras call these.

## Demoable in
Run `/mantra run morning-brief` on a weekday morning — show the full report. Then edit the YAML to add a custom step, re-run, show the change.
